软件设计杂谈：有关接口的两个重要原则

原创 北邮蓝图 北邮蓝图 2020-03-28

 

 

当我们在开发软件时，不可避免的要遇到接口设计的问题。也许是要抽象某一个第三方服务，以便为系统的其他部分提供服务，或者是要设计领域模型的接口模型，又或者是要设计用户接口。

依赖倒置原则（DIP）

 

​x
1.High-level modules should not depend on low-level modules. Both should depend on abstractions (e.g. interfaces).
​
2.Abstractions should not depend on details. Details (concrete implementations) should depend on abstractions.

 

 

依赖倒置原则说：细节应该依赖于抽象。

 

这是什么意思？ [・_・?]

我们先来理解什么是“细节”，什么是“抽象”。在你即将设计的软件系统中，这个软件系统能够为用户带来真正价值的逻辑，被称为业务逻辑，或者“领域”。譬如，聊天软件的价值是信息从用户A传送到用户B，财务管理软件的价值是能够审计，统计财务数据，ATM程序的价值是操作机器存取金钱等。而领域，就是我们要谈论的“抽象”。简单来讲，抽象就是你想做的事情，你想达到的目标。而“细节”与之相对，指的是，为了达到一个目标，我需要怎么做。譬如，对聊天软件的聊天功能，需要先获取用户输入的信息，再通过TCP传送到服务器，服务器再转发到用户B等操作，就属于“细节”。

 

在软件中，“抽象”就是接口，接口就是抽象。而实现就是细节。

 

那么，依赖倒置原则说，“细节应该依赖于抽象”。也就是说，在设计软件系统时，应该先设计接口，再编写其实现，这里的依赖关系体现为“实现”。这是第一层含义。第二层含义是，当一个软件模块的”实现“要调用另一个软件模块时，应该依赖于其抽象，而不是细节。因为”细节应该依赖于抽象”。这里的依赖关系体现为“使用”。

 

这两层含义，十分重要。

 

为什么这么讲呢？一方面，它指导了架构师的工作。因为在软件设计中，第一层含义指导了架构师进行工作的顺序，第二层含义指导了架构师如何设计软件模块或者构件间的关联。另一方面，实施这个原则给软件系统带来了质量的飞跃--允许整个软件系统脱离实际实现的束缚，从而可以呈现出最精华的部分--那些为用户带来价值的逻辑--被清晰的用设计和代码给表示出来--因此获得了非常高的维护性。同时，由于细节依赖于抽象，在不更改软件核心逻辑代码的前提下，我们就可以随意更换数据库，网络协议，线程调度等具体实现。因此，整个软件获得了一个高可移植性，高可扩展性的设计，同时彻底实现了关注点分离，将功能性需求和非功能性需求分开管理，在处理软件用户数量增加等问题上游刃有余--如果软件需要从单体应用扩展为分布式应用，更改某些和执行有关的具体实现就可以了。--类似的非功能性需求，全部都可以找到一个具体的模块或者Aspect（切面）去承担职责--这就是清晰的SRP（单一职责原则）--另一个重要的设计原则，之后会讲到。最后一点是关于架构设计的：这个原则可以帮助我们在设计的某些部分符合整个架构的模式。这句话十分抽象，但我准备用一个简单的例子来说明：假设我们需要让表示领域的代码独立与其它所有代码，这是我们的架构目标。但是领域中有部分逻辑涉及到数据库。那我们是不是就要让领域代码依赖于数据库操作了呢（sql，nosql，orm etc.）？这样就破坏了我们的架构设计目标。此时我们就可以使用DIP，反转这两者的依赖关系。可以参考领域驱动设计中的“Repository模式”，PoEAA中的“领域模型模式”。

 

为什么这个原则如此有效？除了上述有关好处的分析，我们来思考一下它的根源是什么。实际上，抽象是比细节更稳定的东西，而一个抽象可以对应到很多个实现，一个实现却只对应到一个抽象。因此抽象的这种灵活性，实际上就是这样设计系统非常灵活的根本原因之一。同时，修改实现时，依赖于他的代码都要重新编写或者编译或者部署，而依赖于接口的代码则没有这种限制。这是另一种灵活性。

这个原则有更细致的指导[Robert C. Martin 2018]:

▶ 应在代码中多使用抽象接口，尽量避免使用那些多变的具体实现类。

▶ 不要在具体类上创建衍生类

▶ 不要覆盖包含具体实现的函数

▶ 应避免在代码中写入与任何具体实现相关的名字，或者是其他容易变动的事物的名字。

你也许好奇它为什么叫依赖“倒置”：因为控制流和源代码的依赖关系在这个依赖关系处是相反的。

如果你还是不能理解，记住这个原则，总有一天你会顿悟。

 

“细节应该依赖于抽象”。

释意接口（Intention-Revealing Interface）

 

 

xxxxxxxxxx
Name classes and operations to describe their effect and purpose, without reference to the means by which they do what theypromise. This relieves the client developer of the need to understand the internals. These names should conform to the UBIQUITOUS LANGUAGE so that team members can quickly infer theirmeaning. Write a test for a behavior before creating it, to force yourthinking into client developer mode

上一条原则是讲接口何时被设计，而这条原则是讲如何设计接口的。

它可以表述如下：

 

“在命名类和操作时要描述它们的效果和目的，而不要表露它们是通过何种方式达到目的的。这样可以使客户开发人员不必去理解内部细节。这些名称应该与UBIQUITOUS LANGUAGE保持一致，以便团队成员可以迅速推断出它们的意义。在创建一个行为之前先为他编写一个测试，这样可以促使你站在客户开发人员的角度上来思考它”

 

这个原则最主要说的是，命名应该表示”效果“和”目的“。比如对于聊天软件，”消息发送“是目的，而”把消息通过网络发送”是手段。在命名类和操作时要描述它们的效果和目的。我想这个表述已经足够清楚。

 

我们还是通过例子来理解：

假设我们有一个车类（Car），我们有一个功能是让车子跑起来。

不符合该原则的一种设计是：Car.rollWheel()

而符合该原则的一种设计是：Car.run()

其实大家可以看出来，符合该原则的设计基本就是把“功能描述”作为了名字。这正是这个原则想要表述的东西。名字应该表示它“做到了什么”（跑起来），而不是“要做什么”（旋转轮子）。

 

再举一个计算的例子，大家可能更能理解：傅里叶展开。假设我们设计了一个Math类，傅里叶展开是一个静态方法：

Math.fourierTransform()

这样的设计就是符合“释意接口”的设计。

而

Math.splitByOddAndEvenThenButterflyOperation()

就是一个反面例子。

 

实际上，单独讨论这个原则，是容易理解的，在未来的一些post中，我会进一步讨论这个原则和纯函数，副作用，断言等现象（同时也是模式）的关系。

 

“名字要表示目的，而不是操作”

总结

 

 

 

 

接口设计在软件系统中有举足轻重的地位，也是一个大难题。以上两个原则可以帮助大家对接口的设计有一个清晰的认识。接下来的一些文章中，我会进一步阐述DIP和架构的关系，以及释意接口和其他柔性设计方法。
